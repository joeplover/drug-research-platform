# 答辩演示脚本

## 1. 演示目标

本次演示重点说明系统如何围绕药物研发文献处理场景，完成“文献导入、知识抽取、检索增强分析、图谱展示、人工复核和结果追溯”的完整辅助流程。  
演示时不强调替代专家判断，而强调提升资料整理效率、增强结构化处理能力和支持研究人员复核。

## 2. 演示前准备

- 后端服务已启动
- 前端页面可正常访问
- MySQL、Redis、Milvus 已连接
- DeepSeek API 已配置
- Ollama 已启动，并已加载 `qwen3-embedding:4b`
- 数据库中存在至少一个可登录账号
- 系统中至少存在一篇已经导入的样例文献

## 3. 建议演示时长

- 总时长建议控制在 8 分钟左右
- 如果老师提问较多，可缩短图谱和异常说明部分

## 4. 演示顺序与讲解词

### 第一步：项目定位说明（约 1 分钟）

可讲内容：

“我的毕业设计题目是药物研发辅助平台。这个系统不是针对单一药物做固定分析，而是面向更通用的药物研发文献处理场景。系统的目标是把分散的文献内容转成可检索、可抽取、可复核、可追溯的结构化信息，帮助研究人员提高资料处理效率。”

### 第二步：登录与注册（约 1 分钟）

操作：

- 打开登录页
- 展示注册入口
- 使用测试账号登录

可讲内容：

“系统首先提供基础的用户管理功能。这里可以完成注册和登录。这样做的目的是让后续的文献处理、任务记录和操作日志有明确的使用主体，也方便扩展多用户科研协作场景。”

### 第三步：仪表盘总览（约 1 分钟）

操作：

- 进入仪表盘
- 展示统计卡片
- 展示健康状态区域

可讲内容：

“登录后先进入仪表盘。这里可以看到当前系统的文献数量、任务数量、待复核指标数量，以及组件运行状态。这里的健康状态不是写死的，而是会实际检查 MySQL、Redis、Milvus 和 Embedding 服务是否可用，这样更符合真实平台场景。”

### 第四步：文献导入与分块状态（约 1 分钟）

操作：

- 打开文献页面
- 展示已导入文献
- 展示分块和向量同步状态

可讲内容：

“文献进入系统后，平台会先做文本切分，再进行后续的向量化和知识抽取。这里可以看到每篇文献的处理状态，比如是否完成向量同步、同步了多少块、同步时间是什么时候。这样做的好处是每篇文献的处理过程都可以追踪。”

### 第五步：指标抽取与人工复核（约 1.5 分钟）

操作：

- 进入抽取页面
- 展示抽取出的结构化指标
- 现场确认一条指标，或驳回一条指标

可讲内容：

“系统会从文献分块中抽取结构化指标，比如结局指标、时间点、变化值等。因为药物研发场景对准确性要求很高，所以我没有把 AI 输出直接当成最终结论，而是设计了人工复核环节。研究人员可以对每条指标进行确认或驳回，并留下复核说明。这样系统就形成了一个‘自动抽取 + 人工把关’的闭环。”

### 第六步：RAG 报告生成与证据回跳（约 1.5 分钟）

操作：

- 进入分析页面
- 输入一个问题
- 勾选“仅确认指标”
- 生成报告
- 点击一条证据定位，跳转到文献块

可讲内容：

“在分析页里，我用的是检索增强生成思路。系统会先从本地文献块和结构化指标中找相关证据，再调用大模型做综合说明。这里我增加了两个很重要的点。第一是‘仅确认指标’选项，这样报告可以只基于人工确认过的数据。第二是证据定位。每条引用都可以回跳到具体文献块，便于老师和研究人员检查结果来源。”

### 第七步：知识图谱展示（约 1 分钟）

操作：

- 打开图谱页
- 展示节点与关系
- 切换复核状态筛选

可讲内容：

“系统还把文献和指标组织成了一个轻量知识图谱。图谱页可以展示文献节点、指标节点和它们之间的关系。这里还能按复核状态筛选，比如只看已确认指标。这样在做研发资料梳理时，研究人员可以更直观看到证据结构。”

### 第八步：全流程工作流说明（约 0.5 分钟）

操作：

- 展示工作流入口或任务结果

可讲内容：

“如果不想逐步操作，也可以走全流程工作流。系统会把文献处理、指标抽取、分析生成这些步骤串起来，形成一个完整处理链路。这部分很适合后续继续扩展成更完整的科研辅助流程。”

## 5. 演示总结词

可讲内容：

“整体来看，这个系统完成了从文献导入、知识抽取、RAG 分析、图谱组织到人工复核的基础平台搭建。它更像一个药物研发辅助平台原型。它的重点不是替代研究人员，而是帮助研究人员更快地整理文献、发现线索、追踪证据和完成复核。”

## 6. 老师可能会问的问题与回答思路

### 问题 1：这个系统是不是只适合某一种药物？

回答思路：

“不是。当前样例数据里可能有某个具体方向的文献，但系统设计本身是平台化的。它的核心能力是文献导入、结构化抽取、检索增强分析和图谱组织，这些能力可以迁移到更一般的药物研发文献场景。”

### 问题 2：AI 结果怎么保证可靠？

回答思路：

“我没有直接把模型输出当最终结论。系统加入了本地知识检索、证据回跳、引用定位和人工复核。也就是说，模型负责辅助整理，人来做最后判断。”

### 问题 3：为什么要接入 Milvus 和 Embedding？

回答思路：

“因为药物研发文献通常比较长，单靠关键词检索不够。接入 Embedding 和 Milvus 以后，系统可以做语义级的相似检索，这样更容易找到和当前问题相关的文献片段，提高报告生成的依据质量。”

### 问题 4：如果外部服务挂掉怎么办？

回答思路：

“系统现在已经能展示组件真实状态。比如 Embedding 服务不可用时，系统会记录降级状态，而不是伪装成功。这一点在演示时也可以展示，说明平台考虑了可观测性和稳定性。”

## 7. 风险应对

- 如果 DeepSeek 接口响应慢，就提前准备一条已经生成好的报告结果页面
- 如果 Ollama 未启动，就重点展示健康检查和降级提示，不强行演示实时 embedding
- 如果 Milvus 临时不可用，就展示文献向量状态、健康状态接口和系统降级说明
- 如果网络不稳定，就优先演示本地已有数据、抽取结果和图谱页面

## 8. 答辩展示建议

- 先讲平台定位，再演示功能，不要一上来就讲代码
- 多强调“证据可追溯、人工可复核、状态可观测”
- 少说“自动做决策”，多说“辅助研究人员整理和分析”
- 每个页面只讲一个重点，避免演示节奏太快
